v0.1.7 is the first release focused squarely on the app itself rather than infrastructure. Accessibility improvements across all UI controls, a shared game registry that makes adding new games significantly cleaner, richer onboarding diagnostics on Windows, and three meaningful bug fixes including a silent updater failure that users would never have known about. The score holds at 88 — this release didn't close any remaining infrastructure gaps, but it meaningfully raised the bar on app quality and user experience.
What v0.1.7 focused on
v0.1.0–v0.1.6 were almost entirely about getting the infrastructure right — CI pipelines, signing, logging, observability, scoped workflows, audits. v0.1.7 is the first release that invested in the app itself. Accessibility, UX polish, a shared architecture for adding games, and fixing subtle bugs that real users would hit. This is the natural next phase — the scaffolding is done, now the product gets better.
Standout additions
Shared game registry ARCHITECTURE
Setup and dashboard component selection now flows through a centralised game registry rather than scattered conditional logic. This is a meaningful architectural improvement — adding a new game used to require touching multiple places; now it's wired through one registry. It directly supports the roadmap goal of shipping more games. This is the kind of refactor that looks boring from the outside but is essential groundwork.
Accessibility pass across all UI controls UX QUALITY
Labeled inputs, explicit button types, and keyboard-friendly interactions across setup and dashboard screens. This matters for two reasons: it makes the app usable for more people, and it signals that you care about quality beyond "does it work with a mouse." Most side projects never get an accessibility pass at all.
Silent updater failures now logged FIXES SILENT FAILURE
When the auto-updater check failed, it was doing so completely silently in the frontend — no log, no indication to the user or developer. This is now caught and logged. Users on networks that block GitHub releases would have had no idea why they weren't seeing updates. Fixed.

What was fixed / shipped
✓
Game registry centralises component dispatch
Setup and dashboard pages now route to game-specific components through one shared registry instead of scattered conditionals. Clean, extensible, and directly enables shipping more games.
✓
Accessibility improvements across all UI controls
Labeled inputs, explicit button types, keyboard navigation. First systematic accessibility pass — the app now works correctly for keyboard-only users.
✓
WSL onboarding polling race condition fixed
Windows setup flow was reporting completion while WSL commands were still running. Polling now stays in-progress until commands actually finish.
✓
Keyboard events no longer stolen by server cards
Enter/Space on a server card was swallowing keyboard events meant for the nested Start/Stop buttons. Fixed — keyboard interaction now behaves correctly.
✓
Silent updater failures now logged
Frontend now catches and logs updater check failures instead of swallowing them silently. Users on restrictive networks will now have something in the log to reference.
✓
Releases now publish as non-draft
Draft releases don't expose the updater JSON endpoint Tauri polls. Switching to non-draft means OTA updates actually reach users running previous versions.
Remaining gaps
Still 2 of 10 games — but the registry makes this easier now
The game registry refactor was the prerequisite for shipping more games cleanly. That work is done. Valheim or Terraria are both simple Docker images that should work with a JSON recipe only. This is now the #1 priority.
No external crash reporting (Sentry)
Local logging and frontend observability are solid. Still no way to know when users hit crashes unless they report manually. Sentry's Tauri integration is a free tier, ~2 hours of work.
No tests for new accessibility or registry code
The accessibility improvements and registry dispatch logic are untested. These are UI-layer concerns so unit testing is harder, but at minimum the registry's component resolution logic should have tests.
Mock-Docker integration tests for command handlers
Still the longest-standing testing gap. The handlers have spans and are now better structured — they're more testable than ever. This is increasingly the most glaring remaining gap.
No Intel macOS build
Apple Silicon only. Low priority given market share trends but still a gap for Intel Mac users.
