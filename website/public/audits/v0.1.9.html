↑ +2 from v0.1.8
v0.1.9 ships the cubelit-cli — a standalone Rust CLI binary that calls cubelit-core directly, sharing the same SQLite database and Docker orchestration as the desktop app. This is the first real-world proof that the v0.1.8 workspace architecture actually works: CliEventSink is a second concrete EventSink implementation, and the CLI shipped cross-platform binaries (Linux, macOS, Windows) on the same day as the desktop release. The same gaps from v0.1.8 remain open — bollard, Sentry, mock tests, more games — but the score rises because the CLI is a meaningful new capability, not just polish.
What v0.1.9 actually proves
v0.1.8 made a promise: that cubelit-core was genuinely transport-agnostic and that the EventSink abstraction would enable new consumers without changing core logic. v0.1.9 cashes that cheque. The CLI crate was added without touching a line of cubelit-core. CliEventSink streams progress to stderr and log lines to stdout. The identifier resolver, recipe seeding, and man page represent CLI polish that goes well beyond a proof-of-concept. The agent start stub also publicly declares the v0.3.0 remote-agent contract — which maps directly to the MCP server idea discussed earlier.
Standout additions
cubelit-cli crate — standalone Rust CLI binary VALIDATES v0.1.8 ARCHITECTURE
A full CLI binary at crates/cli/ with subcommands: server list, install, start, stop, restart, status, remove, logs (follow with Ctrl+C), and agent start (stub). Calls cubelit-core directly — no Tauri, no GUI. Shares the same SQLite database and Docker orchestration as the desktop. Three cross-platform binaries (10.1 MB Linux, 8.57 MB macOS, 8.28 MB Windows) shipped in the v0.1.9 release assets on the same day as the desktop.
CliEventSink — first non-desktop EventSink implementation ABSTRACTION PROVEN
The EventSink trait introduced in v0.1.8 now has two concrete implementations: TauriEventSink (desktop IPC) and CliEventSink (stderr progress / stdout log lines). This is not a trivial milestone — it's direct proof that the abstraction is correct and general enough to serve fundamentally different transports. NoopSink for tests already existed. The next obvious sink is SentryEventSink.
Identifier resolver + first-run recipe seeding CLI POLISH
Full UUID, unique server name, or unique 4-char UUID prefix — ambiguous matches return a validation error rather than silently picking one. First-run recipe seeding from the default recipes directory with --recipes-dir flag and CUBELIT_RECIPES_DIR env var override. This is the kind of UX polish that distinguishes a real CLI tool from a proof-of-concept.
cubelit.1 man page + release pipeline matured DISTRIBUTION
Man page at crates/cli/man/cubelit.1, installs to share/man/man1 for man cubelit. Release workflow now validates 5 version sources (added crates/cli/Cargo.toml to the version-consistency check). New build-cli job in release.yml produces and attaches CLI binaries to every GitHub release automatically.
agent start stub — v0.3.0 contract declared publicly ROADMAP SIGNAL
The agent start subcommand stub establishes the remote-agent seam contract before the implementation exists. This is the architectural foundation for the MCP server concept — the CLI transport validates the pattern, the agent transport comes next.

Remaining gaps — in priority order
No Sentry / crash reporting — now architecturally trivial to add
The EventSink pattern means this is literally just writing a SentryEventSink impl and wiring it in. There is no longer any structural barrier. This has been the longest-standing gap in the project and is now the most embarrassing one to leave open.
No ServerLifecycle / ServerRunner mock tests
The trait seam has existed since v0.1.8. NoopSink exists. The CLI shipped without trait-level tests. This is now three releases where the mocking infrastructure has been in place and unused for testing. It needs to happen in v0.2.0.
bollard stuck at 0.18 — PR #27 still red
Bollard 0.20 has breaking changes in the streaming API. The call sites in cubelit-core need a manual fix. This dep cannot be left at 0.18 indefinitely — security and compatibility risk grows over time.
8 open Dependabot PRs — some duplicated due to pre-workspace config
The tokio and tauri-plugin-updater PRs appearing twice is a Dependabot config artifact from before the workspace split. The config should be updated to target the workspace root only. Then merge the safe ones.
Still 2/10 games
The CLI now makes game servers scriptable and headless. That's a new reason to want more games. Valheim and Terraria are still the obvious next targets — both are simple Docker images with no sidecar requirements.
CLI ships without its own test coverage
The identifier resolver, subcommand routing, and CliEventSink have no automated tests. At minimum the identifier resolution logic (UUID / name / prefix / ambiguous) should have unit tests — it's pure logic with no Docker dependency.
Dead code: stream_container_logs / ServerLogLine
Should be marked #[allow(dead_code)] with a TODO before compiler warnings start appearing as the workspace grows. Low priority but easy to do.
No Intel macOS build
Apple Silicon only for the desktop. The CLI macOS binary may also be ARM-only. Low priority but worth checking.
