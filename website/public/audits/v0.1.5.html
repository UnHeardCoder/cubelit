Standout addition in v0.1.5
Version guard in CI NEW
The release workflow now runs a check-version job before any builds start. It reads Cargo.toml, package.json, and tauri.conf.json, compares all three against the git tag, and hard-fails if anything doesn't match. This makes the v0.1.3 filename mismatch literally impossible to repeat — the pipeline won't even attempt a build if versions are out of sync. This is a professional-grade release engineering practice.
Automated website deploy NEW
A new deploy-website.yml workflow builds the site, pushes a Docker image to GHCR, and SSH-deploys to a VPS on every release tag. The site now tracks releases automatically — no manual deploy step. The website source lives in a website/ directory in the repo, making the whole stack self-contained and auditable.

Remaining gaps
Docker command handlers still untested
create_server, start_server, stop_server, and delete_server have zero test coverage. These are the highest-risk paths in the app — the ones most likely to break when Docker changes or a user hits an edge case. A mock Docker client with happy-path + error-path tests per command is the next meaningful testing milestone.
Still 2 of 10 games available
Feature completeness is the lowest-scoring category by far. Every new game added moves this significantly. Even one more "Available" game (Valheim or Terraria are the simplest Docker images to wire up) would shift the overall score.
No structured log spans
Log levels exist and the filter is correctly set to warn,cubelit=info, but there are no tracing::info! / tracing::error! call sites in the Docker command handlers where failures actually happen. The log file exists but doesn't capture the events you'd actually need to debug a user-reported crash.
Release notes body is still the default template
The releaseBody in release.yml still outputs the generic "See the assets below to download and install Cubelit" text. It doesn't pull from the CHANGELOG. Now that the CHANGELOG is maintained, wiring it into the release body would make the GitHub releases much more informative.
No Intel Mac build
Only aarch64 (Apple Silicon) macOS is built. x86_64 Mac is declining in market share but still a real user base. Low priority but worth noting.
