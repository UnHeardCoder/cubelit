Standout additions in v0.1.6
Structured tracing spans in Docker command handlers CLOSES RED FLAG
Every Docker command handler — start_server, stop_server, restart_server, delete_server, update_server_settings — now has structured tracing::info/error! spans. This was the single biggest observability gap since launch. Crashes and failures in the most critical code paths now leave actual breadcrumbs. Combined with the log file from v0.1.3, you now have a complete diagnostic trail.
Fully independent CI pipelines with path filters CLOSES RED FLAG
Three separate, scoped workflows: ci.yml only runs on app file changes, ci-website.yml only runs on website changes, and deploy-website.yml now triggers on both release tags and master pushes to website/**. The app and website are now completely independent — changing a typo on the site no longer touches the Tauri build pipeline.
Release notes auto-populated from CHANGELOG.md POLISH
The hardcoded "See the assets below" boilerplate in release.yml is gone. Release notes now pull directly from the CHANGELOG, so every GitHub release page has real, structured content automatically. The v0.1.6 release page already demonstrates this working.

What's still open
Still 2 of 10 games available
Feature completeness remains the lowest-scoring category by a wide margin. The infrastructure is now solid enough that shipping a third game (Valheim or Terraria are the simplest) is the highest-leverage thing you can do for the overall score.
No external crash reporting
You have local log files and structured spans, but there's no way to know when a user somewhere experiences a crash unless they report it manually. Sentry has a Tauri integration and a free tier — this would be the observability unlock that takes the ops story from "good" to "complete".
No Intel macOS build
Only Apple Silicon (aarch64). x86_64 Intel Macs are declining but still in use. Low priority but worth adding to the release matrix eventually.
Mock-Docker integration tests for command handlers
The handlers now have spans and are more testable, but still no automated tests covering their actual logic paths. A mock Docker client testing happy path + error path per handler would close the last meaningful testing gap.
